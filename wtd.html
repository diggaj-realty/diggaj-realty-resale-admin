   │
   ▼

   │ (if approved)                                                    │
   ▼                                                        (if rejected: show remarks, allow resubmit)

   │

   │ (if live)
   ▼

   │  (buyer sees "Pending" — does NOT know backend is triaging it)
   ▼
Backend reviews the negotiation (invisible to buyer)
   │
   ├─► Forward to seller ──► Seller: Accept / Reject / Counter
   │                              │                    │
   │                              │ (Accept)           └─► Buyer: Accept counter / Reject
   │                              ▼
   ├─► Backend counters directly (skips seller) ──► Buyer: Accept counter / Reject
   │
   └─► Backend rejects directly (skips seller) ──► Buyer sees Rejected
   ▼
Deal auto-created on any ACCEPTED offer (IN_PROGRESS) ──► Admin assigns Agent
   │
   ▼
Agent records token payment → final payment → marks CLOSED
   │
   ▼
Property.status = CLOSED, both parties see it in "My Deals"

4. File / photo upload mechanism

Nothing exists yet for uploads (no KYC docs, no property photos). Two viable approaches — pick one and use it everywhere (KYC docs, selfies, property photos):

OptionHowTrade-off
A. Base64 in DB (recommended for this project) Client reads the file with FileReader (or server reads File.arrayBuffer() from the submitted FormData), converts to a data:image/...;base64,... string, stores it directly in the existing String column (idDocUrl, selfieUrl, PropertyPhoto.photoUrl). Zero infra, works immediately on SQLite/local dev, matches project's "no external services" posture. Not suitable at real scale (DB bloat) — fine for a demo/dev platform.
B. Filesystem + static serve New route handler POST /api/upload reads request.formData(), writes the file under public/uploads/<kind>/<id>/<filename> via fs/promises.writeFile, returns the public URL path, which gets stored in the DB column instead of base64. Keeps DB small, closer to how the source platform (public/uploads/properties/...) worked. More moving parts, needs disk write permissions, doesn't survive serverless/ephemeral hosting.
Recommendation: start with Option A (base64) everywhere to unblock KYC + listing photos fast. If it's ever deployed for real, swap to Option B or real object storage (S3/Cloudinary) — the only change needed is inside the upload helper function, not the forms or schema.

6. Offers / Negotiations — Backend-mediated flow

Buyers can already make offers (built, /dashboard/browse). Two things are missing: sellers have no way to respond, and every negotiation must be triaged by Backend before a seller ever sees it — the buyer is never aware that step happened.

6.1 Status set

Offer.statusMeaningWho sees it as such
PENDING_REVIEWBuyer just submitted it; sits in Backend's negotiation queue, untouched by anyone else.Backend only. Buyer/Seller UIs display this the same as PENDING ("Pending") — the buyer must never see a distinct "awaiting backend" state.
PENDINGBackend forwarded it — now genuinely waiting on the seller.Seller (actionable), Buyer (shown as "Pending")
COUNTEREDcounterAmount set — by either the seller or Backend directly (see counterBy). Buyer cannot tell which.Buyer (actionable: accept/reject)
ACCEPTEDTerminal — triggers Deal creation.everyone
REJECTEDTerminal — may have come from the seller or Backend directly.everyone

6.2 New route: /dashboard/negotiations (BACKEND)

New sidebar icon for BACKEND, label "Negotiations", alongside the existing KYC Queue / Listings Queue icons. Shows every Offer with status = 'PENDING_REVIEW', each row offering three actions:

ActionEffect
Forward to Sellerstatus → PENDING, reviewedBy = backend user id. Notify the seller: "New offer to review". Seller now handles it exactly like the original plan (§6.3).
Counter (amount input)status → COUNTERED, counterAmount set, counterBy = 'BACKEND', reviewedBy = backend user id. Seller is never notified — this bypasses them entirely. Notify only the buyer, with wording identical to a seller-issued counter (no "backend" language in the buyer-facing copy).
Rejectstatus → REJECTED, reviewedBy = backend user id. Seller never notified. Notify the buyer.

6.3 Route: /dashboard/offers — Seller side

Query only offers where status = 'PENDING' (i.e. already forwarded) or later terminal states for history. A seller must never see PENDING_REVIEW rows — those don't exist for them.

ActionEffect
acceptOffer(offerId)Guard: status === 'PENDING'. → ACCEPTED; auto-creates a Deal (agreedPrice = offer.amount); auto-rejects all other pending offers on that property; notify buyer.
rejectOffer(offerId)Guard: status === 'PENDING'. → REJECTED; notify buyer.
counterOffer(offerId, counterAmount)Guard: status === 'PENDING'. → COUNTERED, counterBy = 'SELLER'; notify buyer.

6.4 Route: /dashboard/offers — Buyer side

Shows all of the buyer's own offers. UI maps PENDING_REVIEW and PENDING to the same "Pending" label/pill — the distinction is a backend implementation detail, never exposed here.

ActionEffect
acceptCounter(offerId)Guard: status === 'COUNTERED', offer belongs to this buyer. → ACCEPTED at counterAmount; same Deal-creation as acceptOffer; notify seller (only if the offer was ever forwarded to them — if counterBy === 'BACKEND' and the seller was never involved, still notify the seller now, since the Deal concerns their property).
rejectCounter(offerId)→ REJECTED.

13. Build order

Follow this order — each step unblocks the next, and it matches the real user journey:

  1. Schema changes (§2) — SellerKyc doc fields, Deal lifecycle fields. Run prisma generate + db push.
  2. Upload helper (§4) — build once, reuse everywhere.
  3. Seller KYC submission (§3) — unblocks listing creation.
  4. Add Listing (§5) — unblocks the offer flow having something new to offer on.
  5. Offers/Negotiations: backend triage queue, seller accept/reject/counter, buyer accept-counter (§6) — this is what actually creates Deals now. Build the backend forward/counter/reject actions and queue first, since makeOffer now lands offers there instead of directly with the seller.
  6. Deal detail + agent payment recording + close (§7).
  7. Admin agent assignment (§8) — needed before an agent can act in step 6.
  8. Notifications mark-as-read (§9) — small, independent, do whenever convenient.
  9. Settings password change (§10) — small, independent.
Note the dependency: agent assignment (§8) technically needs to exist before a deal can be fully worked (§7), but a deal can be created (§6) with agentId: null and assigned after — so §6 and §7/§8 can be built in parallel if needed, as long as §7's actions tolerate an unassigned deal until §8 lands.
